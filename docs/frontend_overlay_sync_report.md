# Frontend Overlay Sync Report

## Problem

AI event metadata reaches the dashboard through MQTT/WebSocket while video reaches the browser through RTSP/WebRTC/HLS. If the UI renders the newest payload immediately, the box or alert can describe a frame that is not currently visible.

## Previous Flow

`useAiEvents` parsed the incoming alert payload and immediately inserted it into the active AI event feed. The feed reducer still deduplicated alerts, but there was no bounded per-camera overlay buffer and no frontend receive timestamp.

## Improved Flow

The frontend now records `receivedAtMs` as soon as a payload arrives, then computes:

- `networkLatencyMs = receivedAtMs - publishedAtMs`
- `endToEndLatencyMs = receivedAtMs - capturedAtMs`
- `overlayTimestampDeltaMs = abs(selected payload timestamp - target overlay time)`
- `selectedOverlayAgeMs = receivedAtMs now - selected payload receivedAtMs`

Payloads are stored in an `OverlaySyncBuffer` keyed by `cameraLoginId`, falling back to `cameraId`. The buffer is bounded by max age and max size. Overlay payloads are kept in `overlayEvents`, while confirmed incident payloads remain in the existing alert `events` feed so high-rate bbox messages do not create alert cards or trigger acknowledgement flows. Defaults are:

- `VITE_FRONT_OVERLAY_DELAY_MS=300`
- `VITE_FRONT_OVERLAY_MAX_BUFFER_AGE_MS=5000`
- `VITE_FRONT_OVERLAY_MAX_BUFFER_SIZE=300`
- `VITE_FRONT_OVERLAY_MATCH_THRESHOLD_MS=200`

When a payload arrives, the UI selects the buffered payload nearest to `Date.now() - overlayDelayMs`. If a frame id is available to the caller, the buffer supports frame-id nearest matching first; otherwise it falls back to timestamp matching using `capturedAtMs`, then `receivedAtMs`.

`DashboardHomeView` passes `overlayEvents` to `LiveCameraGrid`. Each camera card selects the matching camera event and renders it through `CameraAiOverlay` above WebRTC, HLS, or MJPEG stream content. The overlay renderer accepts `bbox: {x,y,width,height}`, `boundingBox`, `bbox: [x1,y1,x2,y2]`, or `x1/y1/x2/y2` payload shapes, using `frameWidth` and `frameHeight` when available.

## Debug Values

When `VITE_FRONT_OVERLAY_SYNC_DEBUG=true`, alert labels and console logs include frame and latency values:

- `frameId`
- `capturedAtMs`
- `publishedAtMs`
- `receivedAtMs`
- `networkLatencyMs`
- `endToEndLatencyMs`
- `selectedOverlayAgeMs`
- `overlayTimestampDeltaMs`
- LSTM `sequenceStartFrameId~sequenceEndFrameId` or `sequenceStartAtMs~sequenceEndAtMs`

If `overlayTimestampDeltaMs` exceeds the threshold, the hook emits a warning log even when debug mode is off.

## Clock Offset

`capturedAtMs` is generated by the AI host and `receivedAtMs` is generated by the browser. If the two machines are not time-synchronized, direct epoch comparison can be misleading. The current implementation still works as a delay buffer because it can fall back to receive time, but precise frame matching should later estimate a median clock offset from `publishedAtMs` and `receivedAtMs`.

## Verification

Recommended checks:

- Confirm incoming payload logs contain `receivedAtMs`, `networkLatencyMs`, and `endToEndLatencyMs`.
- Confirm `overlayTimestampDeltaMs` appears in debug mode.
- Confirm warnings appear when `overlayTimestampDeltaMs > 200ms`.
- Confirm the active alert UI still shows existing danger cards and acknowledgement controls.
- Compare visible motion and alert timing in WebRTC mode with debug mode enabled.

Current local verification:

- `npx tsc --noEmit`: passed
- `npm run typecheck`: passed
- `npm run test:overlay-sync`: passed, including static contract checks, runtime `OverlaySyncBuffer` behavior checks, AI payload parsing checks, and bbox geometry normalization checks
- `npm run build`: passed
- `git diff --check`: passed, with LF/CRLF warnings only
- Vite dev server `http://127.0.0.1:5173`: returned HTTP 200
- `npm test`: not available because `package.json` has no `test` script
- `npm run test:ai-ack`: passed

Runtime drift verification still requires a running backend, WebSocket alert feed, and WebRTC/HLS stream with `VITE_FRONT_OVERLAY_SYNC_DEBUG=true`. An in-app browser check was attempted, but the browser runtime failed before navigation with a sandbox metadata error, so visual bbox drift has not been proven in-browser yet.

## Follow-Up

- Use browser video timing from `requestVideoFrameCallback` as the direct selection target once the video path exposes an epoch-aligned frame timestamp.
- Add median clock-offset correction for AI host versus browser host.
- Validate across multiple cameras under network jitter.
- Keep debug disabled in production unless investigating overlay drift.
